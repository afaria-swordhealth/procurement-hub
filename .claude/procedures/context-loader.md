# Context Loader

Progressive disclosure rules for context files. Called by `/warm-up`, `/mail-scan`, and any skill that reads project context.

## Motivation

Full context load is ~40k tokens across 4 projects. Most tasks touch one project. A blanket "load everything" wastes 30k tokens and degrades reasoning quality with noise.

Progressive disclosure loads only what the current task needs.

## Layers

### Layer 1 — Index (always loaded)

File: `context/index.json` (generated by `/wrap-up` Phase 2 alongside context sync).

Schema:
```json
{
  "generated": "2026-04-20T23:15",
  "projects": {
    "pulse": {
      "path": "context/pulse/suppliers.md",
      "last_synced": "2026-04-20T23:15",
      "supplier_count_active": 4,
      "supplier_count_rejected": 10,
      "active_suppliers": ["Transtek Medical", "Unique Scales", "Urion Technology"],
      "blocker_count": 1,
      "top_deadline": "2026-04-22"
    },
    "kaia": { ... },
    "mband": { ... },
    "bloompod": { ... }
  }
}
```

Size: ~500 tokens. Loaded at session start.

### Layer 2 — Project context (loaded on demand)

File: `context/{project}/suppliers.md`.

Load when:
- User message names the project explicitly ("Pulse", "Kaia", "M-Band", "BloomPod")
- User message names a supplier in the project's `active_suppliers` list (from index)
- Skill scope is project-bound (e.g., `/mail-scan pulse`, `/meeting-prep` for a pulse supplier)
- `/warm-up --full` was invoked (loads all four)

Do NOT load when:
- Question is ISC-level ("Jorge 1:1 agenda", "team org", "my start date")
- Question is cross-cutting ("weekly report", "all chases", "cron state")
- Skill is session-infrastructure (`session-doctor`, `/ping`)

### Layer 3 — Deep context (explicit opt-in only)

Files loaded only by explicit directive:
- `.claude/knowledge/**/*.md` — Sword Insighter (8 files, ~12k tokens)
- Historical change-log beyond today (git log)
- `outputs/autonomy-ledger.md` full history

Rule: `/ask` is the right way to pull from these. Don't auto-load.

## Resolution logic

Given a user task, derive `required_projects` set:

1. Parse the user message for project mentions (case-insensitive).
2. If none: parse for supplier names. For each match, look up which project's `active_suppliers` contains it.
3. If still empty: check session-state.md `ACTIVE_PROJECT` env-var (set by SessionStart hook) as fallback.
4. If still empty AND the skill requires a project context to operate: ask the user.
5. If the skill is ISC-level or cross-cutting: `required_projects = []` (skip Layer 2).

Load one Layer 2 file per project in `required_projects`.

## Dense format specification

Each Layer 2 file follows this structure (≤200 lines per project file):

```markdown
# {Project} Suppliers — Context File
# Last synced: {YYYY-MM-DDTHH:MM}
# Schema: v1

## Active ({N})

### {Supplier Name}
- status: {Identified|Contacted|RFQ Sent|Quote Received|Shortlisted|PO Issued|Closed|Blocked|Rejected}
- nda: {Not Required|Not Started|In Progress|Sent|Signed}
- currency: {USD|EUR|CNY|PLN}
- unit_cost: {number or null}
- tooling_cost: {number or null}
- last_outreach: {YYYY-MM-DD or null}
- open_ois: {integer}
- next: {one-line next expected action or "awaiting reply"}
- blocker: {one-line blocker or null}
- notes: |
  {Free-form prose. Multi-line OK. Historical detail, audit markers, URL refs.
  This is the existing human-authored body; do not truncate without approval.}

### {Next supplier}
...

## Rejected ({N})
- {list of names, one line each, optional one-line reason}

## Key Decisions
- {bullet list of decisions with dates}

## Open Actions
- {bullet list of ISC-level open actions with owners}
```

Rationale:
- **Structured header** (status / nda / currency / costs / next / blocker) — machine-parseable, drives index.json generation.
- **Notes block** — preserves existing prose. No data loss during migration.
- **Separate sections for Rejected and Key Decisions** — compact, append-only, easy to scan.

Parser: any markdown processor can extract the structured fields by reading bullet lines matching `^- (status|nda|currency|unit_cost|tooling_cost|last_outreach|open_ois|next|blocker):\s*(.+)$`.

## Migration

Existing files (pulse/kaia/mband) use prose-per-supplier. Migration is a **pair-work task** with André (per improvement-plan.md §4):
1. For each supplier, parse existing prose to extract structured fields.
2. André validates each extraction (Notion is authoritative).
3. Emit the dense block; preserve prose in the `notes:` field.
4. Update `# Schema: v1` header marker.

Until migration, `context-loader.md` treats v0 files (no Schema header) as Layer 2 loads but does not attempt to extract structured fields — readers fall back to full-file read.

## Consumers

- `/warm-up` Light mode reads Layer 1 only; Full mode loads Layer 1 + relevant Layer 2 files per the `required_projects` set.
- `/mail-scan`, `/meeting-prep`, etc. — load Layer 2 for their target project.
- `context-doctor` reads the structured fields for schema validation.
- `/ask` does not use this loader — `/ask` has its own pre-built embeddings index.

## Failure modes

- `index.json` missing or stale: `/warm-up` Full regenerates it as part of Phase 2 context sync. Light mode warns and falls back to per-file load.
- Schema version mismatch (file is v1 but parser expects v2): load as free-form; log one-line warning.
- Supplier appears in two projects' `active_suppliers`: the parser picks both projects — correct behavior for cross-project suppliers (e.g., Xinrui on Pulse + M-Band).

## Out of scope

- This loader does not query Notion. `context-doctor` does that.
- This loader does not modify files. It's a read-side rule; writes go through `/wrap-up` and `context-doctor`.
