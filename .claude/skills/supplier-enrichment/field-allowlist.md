# Supplier Enrichment — Field Allowlist (DRAFT, pending André approval)

Generated by first-run pair-work discovery on 2026-04-20. Do not use for live writes until André confirms.

## TL;DR — Critical finding

The four Supplier DBs expose **structured DB properties** (Name, Status, Website, Region, NDA Status, Contact text field, pricing fields, device-specific selects) and a **free-text page body** where `## Profile` (sometimes `## Company`, `## Capabilities`) currently lives as prose. The SKILL.md §Step 1 target table (Legal entity, HQ address, Country, Founded year, Employee range, Parent company, ISO certifications, FDA registration, CE marking, Website URL) **does not map cleanly to structured fields** in any of the four DBs except `Website` (url) and a partial signal via `Region` (select) and `FDA` (Pulse-only select).

André must decide before first live run:

- **Option A — Write to structured fields only.** Restrict enrichment to the handful of typed DB properties that exist (Website, Region, FDA for Pulse). Everything else (legal entity, HQ address, founded year, parent company, employee range, ISO certs) is out of scope until new DB columns are added.
- **Option B — Write to the `## Profile` block in page body.** Replace or append the `## Profile` section with a structured prose paragraph + bullet list generated from the enrichment data card. This preserves the current human-readable format.
- **Option C — Schema extension.** Add typed columns to each DB (e.g., `Legal Entity` RICH_TEXT, `HQ Address` RICH_TEXT, `Founded Year` NUMBER, `Employee Range` SELECT, `Parent Company` RICH_TEXT, `ISO Certifications` MULTI_SELECT, `FDA Registration #` RICH_TEXT) and then enrich into them. Largest schema change; best long-term for queries and Milestone 4 Notion-interface optimization.

Recommendation: **Option B for first run (low risk, reversible, matches current convention), then evaluate Option C as a separate structural sprint.**

### Inconsistencies André must resolve

1. **No structured Profile fields exist today.** The SKILL.md target table was drafted aspirationally. Options A/B/C above are the real choice.
2. **Region vs Country mismatch.** Kaia has "Other" instead of Japan/Taiwan. Pulse has no Taiwan. If we enrich `Country`, we can only map to the coarse `Region` bucket.
3. **FDA handling is Pulse-specific and uses selects, not text.** The SKILL.md target wants a text field for the K-number / establishment number. Today we can only enrich the coarse Cleared/Listed/Pending/N/A status. Real K-numbers live in `Notes` today (e.g., "FDA DXN K231050").
4. **CE marking has no home.** M-Band and BloomPod are the expected CE-relevant projects per the SKILL.md note, but neither DB has a CE field.
5. **Parent company has no home.** Currently buried in `Notes` prose when mentioned at all.

```
## Profile
**Legal entity:** {name}
**HQ:** {address}
**Founded:** {year}
**Employees:** {range}
**Parent:** {parent or "independent"}
**Certifications:** {ISO list or "none found"}
**FDA (Pulse only):** {establishment # or K-number or "not applicable"}
**CE marking (M-Band / BloomPod):** {yes/no/not applicable}
**Notes:** {1-2 sentence manufacturing profile}
```

Write path: `notion-update-page` replacing the `## Profile` section only. Never touches ## Contact, ## Quote, ## Outreach, ## Open Items.

### Phase 3 — Schema extension (Option C, deferred)

Propose a structural sprint to add typed columns. Until then, Phase 2 gives André the same information in a structured prose format.

## Blockers encountered

- **BloomPod DB query rate-limited (HTTP 429).** Schema was fetched successfully via the initial `notion-fetch` on the collection. The `SELECT LIMIT 5` query kept returning 429 after multiple retries. Not a blocker for the allowlist since schema came through, and context/bloompod/suppliers.md confirms the DB has 0 rows.
- **Rejected-query column name quirk.** Pulse initial query failed on `"Last Outreach Date:start"` — the correct SQLite column name is `"date:Last Outreach Date:start"` (the `date:` prefix is mandatory per the SQLite table definition). Worth noting for any future skill using this pattern.
- **No reliable "rejected date" field.** Rejections are mentioned in `Notes` prose, not a structured date. Any future "stability of rejection" logic will need to parse Notes or reference `Last Outreach Date` as a lower bound.
