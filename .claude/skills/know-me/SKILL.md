# Know Me

Scans all memory files and conversation patterns to build a structured profile of how Andre works, decides, and communicates. Other skills reference this profile to tailor their output.

## Pre-flight

1. Read `outputs/session-state.md` for freshness.
2. List all files in `C:\Users\SWORD\.claude\projects\C--Users-SWORD-Documents-GitHub-procurement-hub\memory\`.
3. Read `memory/MEMORY.md` for the index of all entries.

## Step 1: Scan memory files

Read every file in the memory directory. Classify by prefix:

| Prefix | Extract |
|--------|---------|
| `user_*` | Position, tenure, reporting lines |
| `feedback_*` | What went wrong, what Andre wants instead, preference strength |
| `project_*` | Dependencies, constraints, relationships |
| `reference_*` | Static data (addresses, tools) |

For each `feedback_*` file, note the trigger, the correction, and how strong the preference is (one-time vs repeated).

## Step 2: Analyze interaction patterns

Review observable patterns from conversation and memory:

- **Approves fast:** outreach entries, status syncs, formatting fixes.
- **Scrutinizes:** email drafts, pricing, NDA decisions, Rejected status.
- **Overrides:** tone, detail level, cascading suggestions.
- **Signals:** short "yes/go" = move fast. Detailed correction = log to memory. Portuguese = urgent or PT contact.
- **Frustrations:** walls of text, cascading optimizations, repeating known info, raw URLs, "no rush" to suppliers.

## Step 3: Build structured profile

Compile into these sections:

**Role and Context.** Position, start date, reporting lines, 3 projects, 26 suppliers, single operator, relay role on regulatory.

**Decision Style.** Data-driven (tables, metrics). Fast on clear items. Deliberate on supplier-facing or status changes. Delegates: regulatory to Sofia/Bianca, legal to Bradley, BLE to Pedro, Kaia gated on Caio/Max.

**Communication Preferences.** Language per contact (Jorge/Sofia in PT, CN in simple EN, PT suppliers in PT). Tables over prose, short over long. Email: bold headers, reasoning in questions, action closings. Sign-off: "Best," or "Thanks," only. HTML drafts, embedded links, no CDATA.

**Expertise Map.** Andre decides: sourcing, cost analysis, logistics, vendor management, prioritization. Andre relays: regulatory (Sofia/Bianca), legal (Bradley), BLE testing (Pedro), engineering (Miguel/Gustavo), Kaia program (Caio/Max).

**Workflow Preferences.** Likes: structured tables, parallel execution, concise summaries, proactive risk flags. Dislikes: verbose prose, unprompted optimizations, cascading suggestions, repeated info, local paths in Notion.

**Relationship Map.** Key contacts with language, channel, and relationship. Built from CLAUDE.md section 2 + memory files.

**Feedback Patterns.** Table of all rules from `feedback_*` files with source path and strength (Absolute / Strong / Reference).

## Step 4: Save profile

Write to: `memory/user_profile_structured.md`

Header format:
```
# Andre Faria - Operator Profile
# Auto-generated by know-me skill. Last updated: YYYY-MM-DD
# Edit manually or re-run /know-me to refresh.
```

**SHOW BEFORE WRITE.** Present full profile to Andre before saving.

## Step 5: Present for validation

Show profile section by section. Ask:
- "Anything wrong or missing?"
- "Any preferences weighted too strongly or weakly?"

Apply corrections immediately.

## Step 6: Ongoing updates

On future sessions:
1. Check if `user_profile_structured.md` exists.
2. Compare current session behavior against the profile.
3. If a new pattern appears 2+ times, propose an update with rationale.
4. Do not propose updates for one-off corrections.

Loading rule: read only the relevant section, not the full profile. Feedback Patterns is the one section every skill should check.

## Rules

- Focus on collaboration, not evaluation. No negative assessments.
- Frame everything as "how to serve Andre better."
- SHOW BEFORE WRITE for the profile file.
- Profile lives in the memory directory, not the repo. Personal, not shared.
- Do not store sensitive data (passwords, tokens, personal details beyond work context).
- If Andre declines a question, respect it. Leave that section sparse.
- Re-running overwrites the profile. Git tracks history if needed.
